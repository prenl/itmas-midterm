# Multi-Agent Architecture Section for Paper

## 1. Multi-Agent Framing of the System

To align the project with the domain of multi-agent systems, the proposed export recommendation platform is modeled as a specialized multi-agent analytical environment rather than as a single monolithic program.

In this architecture, several autonomous but cooperating agents perform different roles in the overall decision-support workflow. Each agent is responsible for a specific analytical function and exchanges structured intermediate results with other agents through a coordinator.

The system therefore satisfies the main conceptual properties of a multi-agent system:

- functional specialization of agents;
- partial autonomy of individual agents;
- structured information exchange between agents;
- coordinated production of a final global result.

In the present MVP, the agents are implemented as specialized software modules within one execution environment. This is sufficient for a valid multi-agent interpretation because the key defining feature is division of reasoning and task responsibility across interacting agents, not necessarily physical distribution across machines.

## 2. Motivation for Using a Multi-Agent Architecture

The trade recommendation problem addressed in this project is inherently heterogeneous. It combines:

- trade data ingestion;
- macroeconomic data retrieval;
- country-pair contextual analysis;
- export-upgrade opportunity detection;
- recommendation explanation;
- recommendation validation.

A single block of code can technically execute all these tasks, but such an approach is less transparent and less suitable for modular research design. A multi-agent architecture provides several advantages:

- clearer decomposition of analytical responsibilities;
- easier extension of the system with additional reasoning modules;
- better explainability of how recommendations are formed;
- a natural path toward future LLM-based or negotiation-based agents.

For a scientific or academic setting, this architecture is especially useful because it makes the system easier to describe, justify, and experimentally extend.

## 3. Agent Roles

The final implementation includes four specialized agents and one coordinator.

### 3.1 Data Acquisition Agent

The `Data Acquisition Agent` is responsible for obtaining and standardizing the data required by the system.

Its responsibilities include:

- loading Kazakhstan export data from `UN Comtrade` files;
- retrieving macroeconomic indicators from the `World Bank API`;
- reading bilateral country-pair information from `CEPII Gravity`;
- converting heterogeneous raw inputs into a unified structured state.

This agent acts as the interface between external data sources and the internal analytical system.

### 3.2 Opportunity Analysis Agent

The `Opportunity Analysis Agent` is responsible for the core analytical logic of the system.

Its responsibilities include:

- summarizing Kazakhstan’s export base at the `HS6` level;
- linking base exports to downstream processing opportunities;
- ranking candidate upgraded products;
- identifying promising export destinations;
- producing scenario-based export uplift and GDP uplift estimates.

This agent generates the central recommendation outputs of the system.

### 3.3 Economic Explanation Agent

The `Economic Explanation Agent` is responsible for transforming raw analytical results into interpretable economic statements.

Its responsibilities include:

- converting recommendation rows into concise textual explanations;
- describing why a downstream product is economically plausible;
- linking the proposed upgrade to Kazakhstan’s existing export structure and destination markets.

This agent is especially important for explainability and user-facing decision support.

### 3.4 Critic Agent

The `Critic Agent` acts as an internal reviewer of the recommendations generated by the analysis agent.

Its responsibilities include:

- identifying recommendations with weak support;
- attaching risk flags to ambitious or uncertain recommendations;
- distinguishing between more robust and more cautious suggestions;
- reducing the likelihood of overconfident interpretation.

The critic agent improves internal consistency and makes the overall system more credible from an analytical perspective.

### 3.5 Coordinator

The `Coordinator` is responsible for orchestrating the interaction of the other agents.

Its responsibilities include:

- running agents in the correct order;
- maintaining shared state between agents;
- collecting final outputs;
- exporting CSV files, charts, and summary artifacts;
- composing a final system-level summary.

The coordinator does not replace the work of individual agents. Instead, it acts as the control layer that ensures the entire multi-agent workflow proceeds coherently.

## 4. Agent Interaction Logic

The interaction pattern in the implemented system is sequential and cooperative.

The flow is as follows:

1. The `Data Acquisition Agent` collects and normalizes the data.
2. The `Opportunity Analysis Agent` calculates recommendations and market opportunities.
3. The `Economic Explanation Agent` interprets the recommendation set.
4. The `Critic Agent` reviews the same recommendation set and attaches validation comments.
5. The `Coordinator` merges the outputs and writes the final artifacts.

This interaction pattern can be described as a cooperative multi-agent pipeline.

Although the current version is sequential, it is still valid as a multi-agent system because:

- each agent has a distinct role;
- each agent consumes and produces structured state;
- no single agent performs the entire reasoning chain alone;
- the final outcome depends on inter-agent coordination.

## 5. Shared State Representation

The agents communicate through a structured shared state object implemented in the system.

This shared state contains:

- trade rows from Comtrade;
- export-base summary;
- World Bank macro features;
- CEPII gravity features;
- recommended upgrades;
- target-market rankings;
- GDP scenario estimates;
- explanation rows;
- critic rows;
- coordinator-level summary trace.

Using a shared state object ensures that inter-agent communication is explicit and reproducible. It also makes the architecture easier to formalize in the scientific description of the system.

## 6. Why This Qualifies as a Multi-Agent System

It is important to distinguish between:

- a single program that performs many steps internally;
- a system in which multiple agents with different roles cooperate to solve a task.

In this project, the second interpretation is used.

The architecture qualifies as a multi-agent system for the following reasons:

### 6.1 Role Specialization

Each agent is specialized for one part of the workflow:

- acquisition,
- analysis,
- explanation,
- critique,
- coordination.

### 6.2 Inter-Agent Dependence

The output of one agent becomes the input for another.

Examples:

- the `Opportunity Analysis Agent` depends on the state produced by the `Data Acquisition Agent`;
- the `Economic Explanation Agent` depends on the recommendation set produced by the analysis agent;
- the `Critic Agent` depends on the same recommendation set but performs an independent review.

### 6.3 Coordinated Decision Production

The final result is not simply the output of one module. It is the coordinated product of several agents whose outputs are assembled by the coordinator.

### 6.4 Modular Extendability

New agents can be added without redesigning the entire system.

Possible future agents include:

- an `LLM Explanation Agent`;
- a `Policy Simulation Agent`;
- a `Negotiation Agent` for comparing multiple trade strategies;
- a `Forecasting Agent` for more advanced GDP or export projection.

This modularity is a strong indicator of genuine multi-agent design.

## 7. Relationship to LLM-Based Agents

A key conceptual point is that simply adding one large language model after the analytics stage would not, by itself, make the system a multi-agent system.

For example:

- analytics code + one LLM that summarizes results

is better described as a single-agent analytical pipeline with an explanation layer.

By contrast, in the present architecture:

- the explanation role is isolated as a distinct agent;
- the critique role is isolated as a different agent;
- these agents are coordinated explicitly.

Therefore, the system already has a valid multi-agent structure even if the explanation and critique agents are currently rule-based.

Future work can replace these rule-based agents with LLM-based agents without changing the overall architecture.

## 8. Multi-Agent Outputs

The multi-agent runner produces the following outputs:

- recommendation CSV files;
- target-market CSV files;
- GDP scenario CSV files;
- explanation CSV files;
- critic review CSV files;
- visual charts;
- a coordinator-generated summary report.

These outputs are stored in:

- `outputs/multi_agent/`

This output package demonstrates that the agents do not only exist conceptually. They produce concrete, separable artifacts that can be inspected independently.

## 9. Advantages of the Proposed Multi-Agent Design

The chosen architecture provides several practical and scientific advantages.

### 9.1 Transparency

The division of responsibilities makes the recommendation process easier to interpret and explain.

### 9.2 Robustness

The critic agent adds a second layer of reasoning that reduces overconfident conclusions.

### 9.3 Modularity

Each agent can be improved independently.

For example:

- the `Data Acquisition Agent` can later support direct Comtrade API collection;
- the `Opportunity Analysis Agent` can later incorporate `ECI` and `PCI` more deeply;
- the `Economic Explanation Agent` can later be replaced with an LLM-based reasoning module.

### 9.4 Academic Relevance

For a course or study in multi-agent systems, this architecture provides a much stronger basis than a simple sequential script because it explicitly demonstrates:

- decentralized functional roles;
- shared structured communication;
- coordinated reasoning and output formation.

## 10. Limitations of the Current Multi-Agent Version

The current multi-agent system is still an MVP and has several limitations.

### 10.1 Sequential Orchestration

The agents are executed sequentially, not concurrently.

This is acceptable for an MVP, but future work may introduce asynchronous or parallel agent interaction.

### 10.2 Rule-Based Explanation and Critique

The explanation and critique agents currently use deterministic logic rather than advanced reasoning models.

This keeps the MVP stable and reproducible, but limits linguistic richness and deeper qualitative assessment.

### 10.3 Shared-Memory Execution

The agents currently run in one process and exchange information through shared in-memory state.

This is sufficient for the course framing, but future work may explore distributed or message-based agent communication.

## 11. Suggested Academic Formulation

The following wording can be used in a paper:

`The proposed solution is implemented as a cooperative multi-agent decision-support system. Specialized agents are responsible for data acquisition, opportunity analysis, explanation generation, and recommendation validation. A coordinator agent orchestrates information exchange between these agents and integrates their outputs into final recommendation tables and scenario reports.`

Another concise version:

`Unlike a monolithic analytical pipeline, the proposed architecture decomposes the export recommendation task into multiple interacting agents with distinct responsibilities, thereby aligning the system with the principles of multi-agent systems.`

## 12. Future Multi-Agent Extensions

The most promising future extensions are:

1. replace the explanation agent with an LLM-based agent;
2. replace the critic agent with an LLM-based reviewer or debate agent;
3. allow multiple analysis agents to generate competing recommendation sets;
4. introduce a negotiation or voting mechanism between agents;
5. enable parallel agent execution and asynchronous coordination.

These directions would move the system from the current structured analytical multi-agent implementation toward a more advanced intelligent multi-agent platform.
